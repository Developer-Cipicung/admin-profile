# Administrator Module Documentation

The Administrator Module provides a secure interface for managing the backend system administrators. Unlike the News and Product modules, this module operates entirely behind protected endpoints and handles sensitive authentication data.

## Features Implemented (Phase E)

### 1. Administrator Listing (Phase E1)
- **Data Fetching:** Retrieves administrators from `GET /api/v1/admin/administrators` using the `useAdministrators` hook.
- **UI:** Implements a dual-view responsive layout (`AdministratorTable` for desktop, `AdministratorCard` for mobile screens) and standardizes loading states across the application.
- **Filter Toolbar:** Introduced the generic `FilterToolbar` component to elegantly handle debounced searching and active sorting (by newest, oldest, username, and full name).

### 2. Create Administrator (Phase E2)
- **Form Submission:** Sends JSON payload to `POST /api/v1/admin/administrators`.
- **UI & Validation:** Uses `AdministratorForm` built with `react-hook-form` to capture Username, Full Name, Password, and Password Confirmation. Features strong client-side validation mapping identically to backend requirements.
- **Password Toggle:** Developed the generic `PasswordInput` component supporting a clean show/hide text visibility toggle for secure input fields.
- **Conflict Handling:** Specifically intercepts `409 Conflict` errors to warn the user if a chosen username is already taken.

### 3. Administrator View (Phase E3)
- **In-Memory Display:** Completely bypasses the need for an explicit `GET /api/v1/admin/administrators/:id` network request by securely passing the paginated administrator object directly into the generic `ViewDetailsModal`.
- **Future-Proofing:** Architecturally designed so that if the backend ever limits fields on the list payload, a lazy-fetch hook can seamlessly slide into the component without disturbing the visual presentation.

### 4. Delete Administrator (Phase E4)
- **Protected Deletion:** Reuses the generic `DeleteConfirmationModal` to intercept delete requests.
- **Last Admin Protection:** The `useDeleteAdministrator` hook dynamically processes `403 Forbidden` responses generated by the backend's fail-safe mechanism, clearly informing the user that the last remaining administrator cannot be deleted.
- **Intelligent Pagination:** Safely leverages `handlePaginationAfterDelete` to auto-correct the page offset when deleting the sole record on a trailing page.

### 5. Current User Highlighting (Phase E5)
- **Persistence Mechanism:** Intercepts successful login requests via `AuthContext` to persist the active username in `localStorage`, bridging the gap caused by the backend lacking an `/auth/me` route.
- **Visual Distinction:** Actively checks every rendered row and card against the `currentUser` object. The logged-in administrator receives a soft blue highlight, a `(You)` badge, and an unclickable, greyed-out "Delete" action to physically prevent accidental self-deletion.

## Architectural Notes
- No "Edit Administrator" page exists by design to keep the module lightweight and align with simple authentication constraints.
- Any network operations responding with a `401 Unauthorized` are seamlessly caught by the hook and thrown into the global `ErrorBoundary` for immediate session termination.
